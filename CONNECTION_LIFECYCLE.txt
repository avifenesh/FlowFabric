# Connection Lifecycle Analysis

## ConnectionRequest Flow

### 1. ConnectionRequest Structure (client/types.rs, 137 LOC)

**Configuration Fields:**
- read_from: ReadFrom enum (Primary/PreferReplica/AZAffinity)
- client_name: Optional String
- lib_name: Optional String
- authentication_info: Optional AuthenticationInfo
  - username: Optional
  - password: Optional
  - iam_config: Optional IamAuthenticationConfig
    - cluster_name: String
    - region: String
    - service_type: ServiceType (ElastiCache/MemoryDB)
    - refresh_interval_seconds: Optional
- database_id: i64
- protocol: Optional redis::ProtocolVersion
- tls_mode: Option<TlsMode> (NoTls/InsecureTls/SecureTls)
- addresses: Vec<NodeAddress>
- cluster_mode_enabled: bool
- request_timeout: Option<u32> (milliseconds)
- connection_timeout: Option<u32> (milliseconds, default 2000ms)
- connection_retry_strategy: Optional ConnectionRetryStrategy
  - exponent_base: u32
  - factor: u32
  - number_of_retries: u32
  - jitter_percent: Optional
- periodic_checks: Option<PeriodicCheck> (Enabled/Disabled/ManualInterval)
- pubsub_subscriptions: Optional redis::PubSubSubscriptionInfo
- inflight_requests_limit: Option<u32> (default 1000)
- lazy_connect: bool
- refresh_topology_from_initial_nodes: bool
- root_certs: Vec<Vec<u8>>
- client_cert: Vec<u8>
- client_key: Vec<u8>
- compression_config: Optional CompressionConfig
- tcp_nodelay: bool
- pubsub_reconciliation_interval_ms: Optional<u32>
- read_only: bool

### 2. ConnectionRequest Methods
- `get_connection_timeout()` → Duration (with fallback to 2000ms)

### 3. Authentication Information

**Traditional Auth:**
- username: Optional
- password: Optional

**IAM Auth (AWS):**
- cluster_name: Required
- region: Required
- service_type: ElastiCache or MemoryDB
- refresh_interval: 1s to 12 hours (default 14 minutes)
- Token TTL: 15 minutes
- Automatic refresh at 14 minute mark

## Client Lifecycle (client/mod.rs, 2756 LOC)

### Phase 1: Client Creation

**Entry Point:** Implicit via ConnectionRequest building

**Types:**
- `Client` - Main client (async)
- `StandaloneClient` - Non-cluster variant
- `LazyClient` - Lazy-initialized wrapper
- `ClientWrapper` - Internal enum containing both types

### Phase 2: Connection Initialization

**Function:** `get_or_initialize_client()` → RedisResult<ClientWrapper>

**Steps:**
1. Validate ConnectionRequest parameters
2. Create initial connections to addresses
3. Initialize TLS if configured
4. Authenticate (traditional or IAM)
5. Select database
6. Set client name
7. Configure PubSub if needed

**Retry Logic:**
- Exponential backoff with configurable parameters
- Default: 3 retries
- Jitter support to avoid thundering herd

### Phase 3: Connection Pooling

**Managed by redis-rs:**
- Uses MultiplexedConnection internally
- Connection pooling per node
- Automatic connection reuse
- Max inflight requests: 1000 (configurable)

### Phase 4: Topology Management

**Cluster Mode:**
- Initial topology fetch from provided nodes
- Periodic topology checks (default 60s)
- Automatic cluster slot discovery
- Redirect handling for MOVED/ASK

**Standalone Mode:**
- Direct connection to single node
- No topology management
- Simple failover with reconnection

### Phase 5: Health Checks

**Heartbeat:**
- Duration: 1 second (HEARTBEAT_SLEEP_DURATION)
- Periodic connectivity check
- Automatic reconnection on disconnect

**Periodic Topology Checks:**
- Default interval: 60 seconds (DEFAULT_PERIODIC_TOPOLOGY_CHECKS_INTERVAL)
- Connection checks interval: 3 seconds (CONNECTION_CHECKS_INTERVAL)

### Phase 6: Request Execution

**Flow:**
1. Parse command
2. Extract routing information
3. Apply timeout (default 250ms, DEFAULT_RESPONSE_TIMEOUT)
4. Route to appropriate node/connection
5. Handle redirects (cluster mode)
6. Execute on redis-rs connection
7. Convert response via value_conversion module (3753 LOC)

**Timeout Handling:**
- Global: DEFAULT_RESPONSE_TIMEOUT = 250ms
- Per-request: Can override via cmd.timeout
- Per-connection: connection_timeout = 2000ms

### Phase 7: Reconnection

**ReconnectingConnection (619 LOC):**
- Handles transient disconnects
- Implements exponential backoff
- Tracks ReconnectReason enum
- Automatic recovery

**ReconnectReason Variants:**
- ConnectionDropped
- ConnectionError (specific error types)
- AuthenticationFailed
- Other connection failures

**Backoff Strategy:**
- Exponential: base^retry * factor + jitter
- Max retries: configurable (default 3)
- Jitter: Optional random delay to prevent thundering herd

### Phase 8: IAM Token Refresh

**IAMTokenManager (1008 LOC):**
- AWS credential provider integration
- Token generation via AWS SigV4
- Automatic refresh scheduling
- Default refresh at 14 minute mark (900s default TTL / 2)
- Handles credential expiration

**Token Flow:**
1. Get AWS credentials (from environment/IAM role)
2. Generate AUTH token via AWS SigV4
3. Send AUTH with token
4. Schedule refresh before expiration
5. On expiration, regenerate and resend

### Phase 9: Shutdown

**Client Cleanup:**
- Close all connections
- Cancel pending requests
- Stop heartbeat tasks
- Stop topology check tasks
- Release resources

**Graceful Shutdown:**
- Waits for inflight requests to complete
- Configurable timeout
- Automatic cleanup on drop

## State Management

### Per-Client State:
- Current database_id
- Current client_name
- Current username (for multi-user scenarios)
- Protocol version
- Active subscriptions (if PubSub enabled)

### Per-Connection State:
- Node address
- Connection health status
- Retry count
- Last disconnect time
- Active inflight requests

### Global State:
- Runtime (tokio async runtime)
- Script cache (scripts_container)
- Cluster scan cursors (cluster_scan_container)
- Telemetry/OTel configuration

---

## Connection Modes

### 1. Cluster Mode
- Multiple addresses provided
- Topology-aware routing
- Automatic MOVED/ASK handling
- Slot discovery and caching
- Multi-node concurrent requests

### 2. Standalone Mode
- Single address
- Direct routing
- Failover via reconnection
- Simplified state management

### 3. Lazy Connect Mode
- Connection deferred until first command
- Reduces startup latency
- First command may be slower

### 4. Read-Only Mode
- Only read commands allowed
- Connection flag: read_only: bool
- Command filtering/validation

---

## Concurrency Model

**Thread Safety:**
- Client: Send + Sync
- Uses Arc<RwLock<>> for shared state
- Safe for concurrent access from multiple threads

**Async Runtime:**
- Tokio-based
- Multiplexed connections
- Efficient resource utilization

**Backpressure:**
- Inflight request limit (default 1000)
- Automatic throttling if limit reached
- Configurable per client

---

## Error Handling in Lifecycle

**Connection Errors:**
- RequestErrorType::Disconnect → automatic reconnection
- RequestErrorType::Timeout → retry with backoff
- RequestErrorType::ExecAbort → transaction retry
- RequestErrorType::Unspecified → propagate to caller

**Recovery Strategies:**
- Automatic: Disconnect, Timeout (retried)
- Manual: ExecAbort (caller must decide)
- No recovery: Authentication failures (require restart)
