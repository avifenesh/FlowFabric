## Scenario 3 — append_frame + tail_stream

**PASS (with methodology note).**

Spawned a detached writer task appending 5 frames at 100 ms intervals
via `task.append_frame(frame_type, payload, None)`. A concurrent
reader loop observed the same frames; deduped by frame id across
polls.

First attempt used `task.tail_stream(cursor, block_ms=200,
count_limit=100)` on the shared backend. That blocked the
`append_frame` FCALL round-trip behind the `XREAD BLOCK` holding the
ferriskey mux — surfaced as
`EngineError::Transport { .. Valkey(timed out) }` on `append_frame`.
This is the documented head-of-line behavior called out in
`ff_core::engine_backend::EngineBackend::tail_stream`'s rustdoc
(consumers needing isolation build a dedicated backend for tails).

Switched the reader to `task.read_stream(StreamCursor::Start,
StreamCursor::End, 100)` (non-blocking XRANGE on the mux) and the
concurrency resolved cleanly.

Evidence: `frames_tailed = 5/5` observed within the 5 s deadline.

**DX friction recorded.** For consumers who want concurrent
append + tail on a single `ClaimedTask`, the tail-path mux
serialization is easy to hit and the error surface (`Transport /
timed out` on the unrelated `append_frame`) is misleading. Worth
considering either (a) a `ClaimedTask::tail_stream_detached` helper
that mints a dedicated backend connection for the tail, or (b) a
more explicit error when the mux is demonstrably occupied by a
BLOCK call in the same task. Not a release blocker — documented
behaviour.

## Scenario 4 — list_executions / list_lanes / list_flows / list_suspended

**PASS (on the primary read) + DX finding on `list_lanes`.**

Submitted 3 solo executions to one partition, then called all four
new-in-0.5.0 SDK read-models through `FlowFabricWorker`.

Results:

| Call | Result | Assessment |
|------|--------|------------|
| `list_executions(partition_key, None, 100)` | 3 entries, `next_cursor = None` | PASS — shape correct, cursor correctly `None` because `len < limit` |
| `list_lanes(None, 100)` | 0 entries | DX/gap — see below |
| `list_flows(flow_partition_key, None, 100)` | 0 entries | expected (no flows created) |
| `list_suspended(partition_key, None, 100)` | 0 entries | expected (no suspensions) |

**Finding: `ff:idx:lanes` has no writer path.**
`ValkeyBackend::list_lanes` reads `SMEMBERS ff:idx:lanes` and
decodes each member as a `LaneId`. A grep across the shipped
`crates/ff-script/src/flowfabric.lua` for `SADD` on that key returns
nothing — no FCALL populates the registry. `Server::start` does not
seed it either (config `lanes: vec![..]` is only consumed as the
scanner set). Consumers calling `list_lanes` against a freshly booted
cluster will always get an empty page regardless of how many lanes
the server is managing.

Not a release blocker (the call itself returns cleanly, shape is
typed, pagination contract holds). But `list_lanes` is a
new-in-0.5.0 surface and its practical return value is "always
empty" until an out-of-band `SADD ff:idx:lanes <name>` runs. Likely
candidates:

- `ff_create_execution` or `ff_issue_claim_grant` should `SADD` the
  lane into `ff:idx:lanes` on first sight of a new lane name, OR
- `ff-server` should seed `ff:idx:lanes` from `ServerConfig.lanes`
  at boot.

Filing as a 0.5.1 candidate — worth a dedicated issue.

## Verdict

**v0.5.0 healthy for external consumers.** All 6 runtime scenarios
pass end-to-end + the build smoke. Published artifacts are reachable
from a fresh scratch project with default features; the headline
new-in-0.5.0 surfaces all work (`list_*` read-models, tower-layer
wrap, `EngineBackendLayer`/`TracingLayer`, `ff-observability-http`
router).

One DX finding for 0.5.1 consideration:

1. **`ff:idx:lanes` has no writer.** `FlowFabricWorker::list_lanes`
   always returns 0 entries on a freshly-booted system. Either the
   FCALL layer should `SADD` on lane-first-sight, or `ff-server`
   should seed from `ServerConfig.lanes` at boot. Not a blocker;
   the call itself is shape-correct.

One documented friction confirmed:

2. **Concurrent tail + write on a single `ClaimedTask` hits the
   ferriskey mux head-of-line.** Error surfaces as
   `Transport/timed_out` on the write rather than something more
   actionable. Worth either a helper that mints a dedicated tail
   backend or a clearer error at the `tail_stream` boundary. Already
   documented in the trait rustdoc; not a new regression.
