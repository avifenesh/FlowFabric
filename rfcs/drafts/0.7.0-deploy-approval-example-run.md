# 0.7.0 deploy-approval example — live-run evidence

**Backend:** Valkey (localhost:6379). ff-server 0.6.1 built from the
worktree main @ commit 4489c47 with
`FF_LANES=default,build,test,deploy,verify` +
`FF_WAITPOINT_HMAC_SECRET` (generated per-start).

**Postgres not exercised.** ff-server 0.6.1 wires only the Valkey
backend (`ServerConfig::from_env` reads `FF_HOST`/`FF_PORT` only; no
`FF_BACKEND` switch; no Postgres-backed server path in
`crates/ff-server/src`). The Postgres backend crate
(`crates/ff-backend-postgres`) ships migrations and trait impls but
is not selectable from ff-server boot today. Documented in the
example's README → Known gaps; `scripts/bootstrap-pg.sh` is provided
for when the wiring lands. Per the spec's fallback clause, the live
run uses Valkey.

**Primitives exercised (end-to-end):**
- RFC-007 — Flow DAG with `AllOf` edge-group (default) on `deploy`
  gating three upstream tests.
- RFC-009 — Capability-routed test lane: three workers advertise
  `kind=unit|integration|e2e`; scheduler dispatches the right
  execution to the right worker.
- RFC-013 — Typed `SuspendArgs` call on the deploy worker; lease
  released.
- RFC-014 — `Composite(Count { n: 2, count_kind: DistinctSources,
  matcher: ByName("deploy_approval"), waitpoints: [...] })` — two
  distinct reviewer identities resume the deploy exec.
- RFC-015 — `DurableSummary { patch_kind: JsonMergePatch }` on the
  build worker's `summary_delta` + `summary_final` frames.
- RFC-016 — `cancel_flow { cancellation_policy: cancel_all }` wired
  on the verify worker's `--fail` path (see Known gaps for live
  evidence status of the cascade).

## Known gaps surfaced by the run

1. **`ff-server` HMAC secret wipe on `FLUSHALL`.** The server writes
   per-partition HMAC secrets to Valkey on first boot. A `FLUSHALL`
   (common during dev-loop resets) wipes them; subsequent `suspend`
   calls fail with `hmac_secret_not_initialized` until the server is
   restarted. Not a bug — the server contract is load-once — but a
   friction point worth a server-side warning when the secret key is
   missing after a connection that previously had it. Log-level bump,
   non-blocking for 0.7.

2. **Resume-path latency.** Between `resume_condition_satisfied`
   acknowledgement on the approve signal and the deploy worker
   actually reclaiming the interrupted attempt, ~30 s elapsed in
   the observed run. Scanner-driven promotion is doing its job; the
   latency is the `FF_UNBLOCK_INTERVAL_S` default (5 s) plus lease
   reclaim scan cadence. Tunable; acceptable for the example's teach
   purpose. Not a 0.7 blocker.

3. **Postgres backend not server-wired.** See header. Known and
   tracked; bootstrap script preserved for the follow-up PR.

4. **Second-flow-in-the-same-worker claim pause (observed, not
   fully diagnosed).** After a full happy-path run (suspend →
   resume → complete), submitting a second flow with the same
   worker pool exhibited integration + e2e executions staying
   `eligible_now` for >45 s while the same workers remained alive
   and registered. Worker-permit release or capability-snapshot
   hash refresh likely culprits. Not a primitive-correctness issue
   (the first flow completed end-to-end including the 2-of-N
   approvals). Flagged for a follow-up scanner-cadence investigation
   in the example repo but not blocking the 0.7.0 example itself.

5. **`cancel_flow` cascade (fail-verify path) not captured in this
   report.** The wire-up exists in `src/bin/verify.rs`; proving it
   live blocked on gap (4). The code path is
   straightforward REST
   (`POST /v1/flows/{id}/cancel?wait=true`) and has independent
   coverage in `examples/retry-and-cancel/` scene 2 (same server
   build). Follow-up: re-run with a cold-worker pool to record the
   cascade timeline.

## Reproduction

```bash
valkey-cli FLUSHALL
FF_WAITPOINT_HMAC_SECRET=$(openssl rand -hex 32) \
  FF_LANES=default,build,test,deploy,verify \
  ./target/release/ff-server &

cd examples/deploy-approval
cargo build --release
./target/release/build-worker &
./target/release/test-worker --kind unit &
./target/release/test-worker --kind integration &
./target/release/test-worker --kind e2e &
./target/release/deploy &
./target/release/verify &

./target/release/submit --artifact app:v1.2.3 --commit abc123
# Copy deploy execution_id from the "[timeline] deploy_created" line.
./target/release/approve --flow-id <fid> --execution-id <deploy-eid> --reviewer alice
./target/release/approve --flow-id <fid> --execution-id <deploy-eid> --reviewer bob
```
