## 1. FCALL function inventory

Forty-six FCALL functions across 11 Lua files. Every entry lists declared
KEYS/ARGV counts, the Valkey primitives the body invokes, the atomicity
assumption the function relies on, and a one-line Postgres-translation
difficulty score.

**Atomicity column conventions.**
- `HT {p:N}` â€” all KEYS carry the `{p:N}` partition hash tag (single slot, executes atomically inside the Lua sandbox).
- `HT {p:N} + cross-tag` â€” function accepts a key from a *different* hash tag (e.g. global index) where the contract is "best-effort, idempotent" rather than strict atomic.
- `Single-key` â€” function touches only one key family and atomicity is trivially a single-shard property.
- `FCALL body` means the Lua atomicity guarantee itself (Valkey runs the whole body under the slot lock).

## 2. Shared infrastructure (ff-script)

### 2.1 Loader (`loader.rs`)

`ensure_library(&Client)` is the startup hook that every ff-server and
ff-sdk session runs:

1. `FCALL ff_version 0` â†’ compare with `LIBRARY_VERSION` (compiled-in from `flowfabric_lua_version`).
2. Mismatch or absent â†’ `FUNCTION LOAD REPLACE` with the bundled `LIBRARY_SOURCE` (`flowfabric.lua`, generated by `scripts/gen-ff-script-lua.sh`).
3. Retry `FUNCTION LOAD` up to 3Ă— on transient failures; short-circuit on syntax errors.
4. Re-verify via `ff_version`.

**v0.7 impact.** Entire loader becomes a no-op in a non-Valkey backend:
Lua source ships zero server-side code, so there is nothing to "load".
The equivalent Pg shape would either be migration-checked stored
procedures (if we keep server-side logic) or nothing at all (if all logic
moves client-side into a Pg-native Rust backend). **Recommendation:** drop
the loader abstraction at the trait boundary; make it an
`EngineBackend::ensure_ready()` hook that Valkey implements and Postgres
leaves empty.

### 2.2 Error taxonomy (`error.rs`)

`ScriptError` is the canonical error enum â€” 85 Lua-sourced variants plus
`Valkey(ferriskey::Error)` and `Parse{fcall, execution_id, message}`.

- **Lua wire protocol:** Lua returns `{0, "error_code", ...detail}` on failure, `{1, "OK"|"ALREADY_SATISFIED"|"DUPLICATE", ...fields}` on success (RFC-010 Â§4.9).
- **`ScriptError::from_code(code)`** maps snake_case strings â†’ enum variants.
- **`from_code_with_detail` / `from_code_with_details`** additionally fold Lua detail fields into variants that carry payload (`CapabilityMismatch`, `InvalidInput`, `ExecutionNotActive{...}`, `RotationConflict`, `InvalidTagKey`, `InvalidCapabilities`, `InvalidPolicyJson`).
- **`.class() -> ErrorClass`** classifies every variant as Terminal / Retryable / Cooperative / Informational / Bug / Expected / SoftError (ff-core). This is the SDK's dispatch table for retry vs fail.
- **`engine_error_ext.rs`** provides `EngineError` extensions (e.g. `valkey_kind(&EngineError) -> Option<ErrorKind>`) so ff-core stays transport-free.
- **`retry.rs`** owns the `ferriskey::ErrorKind` â†’ retryable boolean + `kind_to_stable_str` wire-contract mapping (used in HTTP `ErrorBody.kind`).

**v0.7 impact.** The Lua-sourced variants are backend-neutral â€” an
EngineBackend trait implementation over Postgres should raise the *same*
error codes for the same semantic conditions (stale-lease, lease-expired,
rate-limit-exceeded). The `Valkey(ferriskey::Error)` variant is the only
transport-specific one; parallel Pg variants (e.g. `Postgres(sqlx::Error)`)
go behind a feature flag symmetric to `valkey-client`.

### 2.3 FCALL wrappers (`functions/*.rs` + `macros.rs` + `result.rs`)

**`ff_function!` macro** (`macros.rs`) generates each async wrapper:

```rust
async fn $fn_name(conn: &ferriskey::Client, ctx: &$ctx_type, args: &$args_type)
    -> Result<$result_type, ScriptError>
{
    // build Vec<String> of keys + argv via caller-supplied exprs
    let raw = conn.fcall::<Value>(stringify!($fn_name), &keys, &argv).await?;
    <$result_type as FromFcallResult>::from_fcall_result(&raw)
}
```

**Key contexts** are per-domain `OpKeys<'a>` structs (`ExecOpKeys`,
`FlowStructOpKeys`, `DepOpKeys`, `SchedOpKeys`, `SignalOpKeys`,
`SuspendOpKeys`, `ResumeOpKeys`, `WaitpointOpKeys`,
`ExpireSuspensionOpKeys`, `StreamOpKeys`, `QuotaOpKeys`, `BudgetOpKeys`,
`BlockOpKeys`, `ResolveDependencyKeys`, `SetEdgeGroupPolicyKeys`). These
hold the partition prefix + IDs needed to render each KEYS slot.

**ARGV serialization** is uniformly `String::from_str`-style: each ARGV
slot is a `String` built via `.to_string()` on typed fields (IDs, numbers,
enums â†’ their RFC-stable snake_case names). No binary protocol; no
`struct.pack`. JSON-encoded fields (`policy_json`, `tags_json`,
`resume_condition_json`, `tree_json`) are pre-serialized by the caller.

**Result parsers** (`FromFcallResult`) go through `FcallResult::parse` in
`result.rs`, which:
1. Asserts the top-level `Value::Array`,
2. Extracts `{[0]=Int status_code, [1]=status_string, ...tail}`,
3. Hands `(status, &tail)` to the typed parser, which builds a Rust
   enum variant per status string (`OK` / `ALREADY_SATISFIED` /
   `DUPLICATE` for success; error code for failure).

Each result parser that needs to distinguish success variants
(`CompleteExecutionResultPartial`, `CancelExecutionResultPartial`,
`ChangePriorityResultPartial`, `ClaimExecutionResultPartial`,
`ClaimResumedExecutionResultPartial`, `ExpireExecutionResultPartial`,
`MoveToWaitingChildrenResultPartial`, `DelayExecutionResultPartial`)
matches on `status` and decodes the tail.

### 2.4 Direct ferriskey calls beyond FCALL

Only one caller in the crate reaches beyond FCALL dispatch:

**`stream_tail.rs :: xread_block(...)`** â€” issues `XREAD [BLOCK ms] COUNT
n STREAMS key id` + a follow-up `HMGET stream_meta closed_at
closed_reason`. Blocking reads are rejected by `FUNCTION`, so this must
stay outside the Lua library. The `block_ms + 500ms` timeout extension
from ferriskey is documented in the module header. This is the biggest
single "non-FCALL" surface and the main driver for keeping
`XREAD BLOCK` semantics in any Postgres-backed equivalent (e.g.
`LISTEN/NOTIFY` + long-poll select, or a `pgq`-style client wait).

Everything else goes through the `ff_function!`-generated FCALL wrappers
or `Client::function_load_replace` in the loader.

### 2.5 Version / reload logic

`lua/version.lua` returns the literal `'26'`. The contract is enforced by
`scripts/gen-ff-script-lua.sh` extracting the `return 'X'` literal into
`crates/ff-script/src/flowfabric_lua_version`, which
`LIBRARY_VERSION: &str` reads via `include_str!`. CI diffs on drift.

Every ARGV or KEYS arity change requires a bump â€” documented in
`version.lua` itself. In a Postgres backend this file is dead code; its
replacement is the SQL migration version table.

## 3. Postgres-incompatible idioms

**Tier A: trivially equivalent in Postgres.**

- `SET key NX EX ttl` â€” dedup via `INSERT INTO dedup(key) ... ON CONFLICT DO NOTHING` with a scheduled purge or a TTL column.
- `HSET`, `HGETALL`, `HGET`, `HMGET`, `HINCRBY`, `HDEL`, `HKEYS` â€” one-to-one with jsonb / column sets / a key-value child table.
- `SADD`, `SREM`, `SMEMBERS`, `SISMEMBER`, `SCARD` â€” a child-table of `(set_key, member)` with a UNIQUE index.
- `ZADD`, `ZREM`, `ZSCORE`, `ZCARD`, `ZRANGE`, `ZREMRANGEBYSCORE` â€” a child-table of `(zset_key, member, score)` with a B-tree on `(zset_key, score)`. Sliding-window rate limits translate directly.
- `INCR` / `DECR` â€” `UPDATE ... SET n = n Â± 1 RETURNING n` under row lock.
- `EXISTS`, `GET`, `DEL` â€” trivial.
- `PEXPIREAT`, `PEXPIRE`, `PERSIST` â€” maintain an `expires_at` column and a janitor that runs `DELETE WHERE expires_at < now()`. Valkey's strictly timed eviction becomes "eventually consistent" in Pg; schedulers must cope.
- `redis.call("TIME")` â€” replace with `now()` at the SQL level (or caller-threaded `now_ms`). The engine already threads `now_ms` through most ARGV signatures; `TIME` is only invoked for *defensive* time inside the sandbox. Centralizing on a single clock source for v0.7 is a cleanup opportunity.
- `redis.status_reply` / `redis.error_reply` â€” the wire protocol envelope (`{0|1, code, ...fields}`) is entirely our convention and is trivial to reproduce from a Postgres backend.

**Tier B: non-trivial but feasible.**

- **`XADD stream * k v ...` + `XTRIM`/`MAXLEN`** â€” append-only frames table with `(exec_id, attempt_id, frame_id, fields JSONB)` and a retention job trimming by count or by wall clock. `XADD` MAXLEN semantics are probabilistic (`~`); Pg can do exact `DELETE` with a LATERAL JOIN, or approximate via `pg_partman`.
- **`XRANGE from to COUNT n`** â€” `SELECT ... WHERE frame_id BETWEEN from AND to ORDER BY frame_id LIMIT n`. Stream ID (`millis-seq`) sorting is lexicographic on zero-padded prefix or parse-and-order â€” pick the encoding at migration time.
- **`cjson.encode/decode`** â€” Postgres has native `jsonb`. Signal-condition trees can live as `jsonb` directly; the Lua-side evaluator (`helpers.lua`) becomes Rust server code. No semantic loss.
- **`redis.sha1hex` + `hmac_sha1_hex` in `lua/helpers.lua`** â€” implemented by the Lua sandbox; in Pg we move HMAC mint/validate into Rust (already-landed `hmac` crate dependency in ff-core is straightforward). This actually *improves* testability.
- **Multi-key atomicity via `{p:N}` hash tag** â€” Valkey's slot-locked execution inside `FUNCTION` gives us a free transaction. Postgres wraps every FCALL-equivalent in a single `BEGIN ... COMMIT` and gets stricter SERIALIZABLE semantics â€” no translation difficulty beyond "everything is a transaction."

**Tier C: redesign required.**

- **`XREAD BLOCK ms STREAMS ...`** in `stream_tail.rs`. Pg has no native blocking stream tail. Options:
  1. `LISTEN frames_channel` + server-side `NOTIFY` on `INSERT` (short-poll bridge).
  2. Long-polling over a frames-by-id window with backoff.
  3. A dedicated `pgq`-style job polling loop.
  **Impact:** SDK tailing latency characteristics change; explicit design choice required.
- **`PUBLISH ff:dag:completions` + channel fan-out** (used by `ff_complete_execution`, `ff_cancel_execution`, `ff_fail_execution`, `ff_expire_execution`, `ff_cancel_flow`, `ff_resume_execution`, `ff_expire_suspension`). The scheduler/promotion loop relies on these wake-up events to avoid busy-polling blocked-zsets. Pg equivalent is `LISTEN/NOTIFY`, which has queue-depth limits and delivery semantics that diverge from Redis PubSub (no "fanout to N waiters" without per-listener wiring). **Design decision** on whether v0.7 collapses this to a periodic scan or re-implements fanout with `LISTEN`.
- **`FUNCTION LOAD REPLACE` + server-side Lua execution.** Gone entirely. Every Lua body's logic ports to Rust, executed by an `EngineBackend::Postgres` impl over `sqlx::Transaction`. This is the bulk of the migration work â€” the 11 Lua files (~7,000 lines counting helpers) become Rust, sharing types with the engine.
- **Stream ID semantics (`<ms>-<seq>`)** â€” we need a monotonically-increasing ID generator in Pg that survives crashes; simplest is `bigserial` + wall-clock-encoded prefix. Verify that RFC-006's stream-ID ordering assumptions (from / to cursors, strict ordering per key) hold.

**`struct.pack`: not used.** Searched `lua/**`; no binary packing anywhere. The token+frame formats are all string-based. One less migration worry.

## 5. Open questions for owner

1. **PUBLISH fan-out policy.** Postgres `LISTEN/NOTIFY` vs periodic scan vs pgq-style queue â€” which for v0.7? Affects 7 FCALLs' terminal paths plus suspend/resume. Low-risk answer (scan every N ms) adds latency; high-risk answer (NOTIFY) reintroduces a connection-scoped delivery guarantee that can drop under pressure.
2. **XREAD BLOCK translation.** Do we keep the blocking SDK tail (now a `LISTEN`-backed long-poll) or flip SDK consumers to pull-with-backoff? RFC-006 tail consumers assume low-latency wake; any change is a user-visible API contract shift.
3. **Stream ID encoding.** Valkey `<ms>-<seq>` IDs are lex-orderable. Pg alternatives: (a) `(ts_ms bigint, seq int)` composite with `(ts_ms, seq)` index; (b) monotonic `bigserial` + materialized `ms-seq` string; (c) ULID. Impacts migration shims and historical-data export.
4. **HMAC key material home.** Valkey keeps the secrets hash (`ff:sec:{p:N}:waitpoint_hmac`) partition-local; in Pg the natural home is a global secrets table with grace TTL. Confirm: do we keep per-partition HMAC scoping in v0.7, or collapse to one global keyring?
5. **MAXLEN retention semantics.** `XADD MAXLEN ~ N` is approximate; the engine relies on "frames older than the last N are gone" but not strict. Pg equivalent can be strict (count-limited `DELETE`). Do we ship strict semantics (simpler, less IO) or approximate-via-partitioning (closer to Valkey behavior, higher throughput)?
6. **Loader/version negotiation.** When both Valkey and Postgres backends ship, `ff-script::LIBRARY_VERSION` becomes Valkey-only. Does `EngineBackend` expose a `backend_version()` that SDKs pin against, or do we rely on semver of the Rust crate alone? Matters for mixed-version rollouts.
7. **Capability-matching location.** `parse_capability_csv` + `missing_capabilities` live in Lua with tight token/length bounds. In a Pg backend this moves into Rust server code â€” is there a case for moving the whole capability check into a ff-core helper *now* so both backends share the implementation?
