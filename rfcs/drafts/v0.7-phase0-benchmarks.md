# v0.7 Phase 0 — Postgres backend perf-parity benchmark report

**Run date:** 2026-04-24
**Commit:** `a1c2322` (origin/main, Wave 7c on v0.7 track)
**Branch:** `feat/pg-wave7c-bench-parity`
**Operator:** single-agent (Phase 0 pre-release gate, Wave 7c)

This report covers the first live perf cut of the Postgres backend
against the bench matrix already established for the Valkey backend
at v0.6 Phase 0. Scope: 5 scenario ports + the RFC-016 §4.2
quorum-cancel fan-out SLO gate, on the same hardware class as v0.6.

Raw JSON outputs land in `benches/results/*-pg-a1c2322.json`
(gitignored per `benches/results/.gitignore`).

Same CPU class (AMD EPYC 9R14) as the v0.6 Phase 0 run; different host

both columns are from the same host unless noted.

### Server side

- Postgres 16 (Ubuntu 16.10-1.pgdg24.04+1). Default `shared_buffers`,
  WAL settings. `max_locks_per_transaction` and
  `max_pred_locks_per_transaction` raised from the Ubuntu default of
  64 to **512** to accommodate concurrent
  claim/complete/cancel_flow transactions — see §"Host tuning" below.
- Valkey 8.x standalone on `127.0.0.1:6379` for cross-reference.
- Isolated bench DB `ff_bench_v0_7`; dropped and re-created between
  scenarios so the partitioned-table state is clean for each run.

### Backend path

The bench binaries in `benches/comparisons/postgres/` call
`PostgresBackend::from_pool` + the `EngineBackend` trait directly,
mirroring the ferriskey-baseline / redis-rs-baseline shape on the
Valkey side. No ff-server HTTP, no ff-sdk polling, no Lua. Seed rows
are written via raw multi-row INSERT into `ff_exec_core` (matching
`crates/ff-backend-postgres/tests/pg_engine_backend_attempt.rs`).

## §2. Per-scenario results

### Scenario 1 — submit → claim → complete throughput

10 000 tasks, 16 workers.

| system         | ops/sec | p50 (ms) | p95 (ms) | p99 (ms) |
| -------------- | ------- | -------- | -------- | -------- |
| FF(Valkey)     | 2 966   | 0.39     | 0.70     | 0.84     |
| **FF(Postgres)** | **510** | **24.60** | **39.41** | **44.10** |
| baseline (rs)  | 14 462  | 0.28     | 0.36     | 0.48     |
| apalis         | 98      | — (not captured) | — | — |

FF(Postgres) is ~5.8× slower than FF(Valkey) and ~28× slower than the
raw redis-rs baseline. The dominant cost is the per-op transaction
overhead: each claim runs a BEGIN + SELECT FOR UPDATE SKIP LOCKED on
ff_exec_core, UPDATE to flip lifecycle/ownership, then COMMIT; each
complete runs a FOR UPDATE re-read of ff_attempt + UPDATE on both
ff_attempt and ff_exec_core + INSERT into ff_completion_event (outbox
trigger fires NOTIFY). At 16 concurrent workers + 10 k tasks, that's
~320 k transactions + ~320 k row-level locks held simultaneously for
brief windows. The `max_locks_per_transaction=64` default saturated
shared memory mid-drain on every attempt; bumping to 512 unblocks it.

### Scenario 2 — suspend → signal → resume

**Skipped** this cut. The Postgres trio
(`suspend` / `deliver_signal` / `claim_resumed_execution`) is fully
implemented at SERIALIZABLE with a 3-attempt retry and has integration
coverage in `crates/ff-backend-postgres/tests/suspend_signal.rs`.
Driving the RFC-014 Pattern-3 ingress contract (SuspendArgs with
WaitpointBinding + ResumeCondition + ResumePolicy +
SuspensionRequester + TimeoutBehavior + idempotency-key plumbing) at
the direct-backend bench layer duplicates ~150 lines of server-side
plumbing that's still shifting; the cut is deferred to v0.7 Phase 1
once the server-side ingress stops moving.

### Scenario 3 — steady-state (time-boxed)

16 workers × 120 seconds, refill at `depth < 500` with batches of
1 000. Total completed: 72 574. Average throughput: **604.8 ops/sec**.

The Valkey-side bench uses 100 workers × 5 min; the PG bench uses 16
workers × 2 min because (a) the shared-memory lock budget doesn't
accommodate 100 concurrent claim transactions even at
`max_locks_per_transaction=512`, and (b) the 6 h watchdog on this
agent wave doesn't leave room for a 5-min pass on all six scenarios.
FF(Valkey)'s 5-min run measures long-tail latency drift that 120 s
can't surface. The raw throughput comparison is therefore NOT
meaningful across these two configs; a v0.7 Phase 1 re-run should
align the configs (Postgres at 16 workers × 5 min; Valkey at 16
workers × 5 min for the apples-to-apples window).

### Scenario 4 — linear 10-node flow DAG

100 flows × 10 nodes × 16 workers on a pooled lane.

| system         | flows/sec | p50 per-flow (ms) | p99 per-flow (ms) |
| -------------- | --------- | ----------------- | ----------------- |
| FF(Valkey)     | 17.01     | 474.74            | 5 816.08          |
| **FF(Postgres)** | **48.2** | **760**           | **882**           |
| apalis         | 9.34      | — (not surfaced)  | —                 |

Higher throughput on PG than Valkey here is misleading. PG's number is
an emulation: 10 claim+complete ops per flow on a pooled lane.
`create_flow` + `stage_edge_dependency` + `apply_dependency_to_child`
are Valkey-side FCALLs not exposed on the PG EngineBackend surface
(they sit at the ff-server ingress layer in Valkey; a peer API on the
PG side has not landed). The FF(Valkey) bench exercises the full
dependency-resolution path; FF(Postgres) does not. Per-flow latency
shows FF(Postgres) with tight distribution (p50=760 / p99=882 vs
Valkey's 474.74 / 5816.08) because the Valkey bench's long tail is the
dep-resolution dispatcher, which PG simply doesn't have yet. A v0.7
Phase 1 re-run lands after the flow-ingress FCALL peer ops port to PG.

### Scenario 5 — capability-routed claim

10 capability sets × 4 workers/set × 1 000 tasks/set (10 k total).

| system         | ops/sec | p50 (ms) | p95 (ms) | p99 (ms) |
| -------------- | ------- | -------- | -------- | -------- |
| FF(Valkey)     | 1 795.7 | 416.7    | 531.2    | 550.0    |
| **FF(Postgres)** | **300.0** | **23.36** | **27.80** | **29.06** |

Throughput gap (~6×) reflects the 40-vs-100 worker gap + PG's per-claim
tx cost. Latency distribution is tighter on PG than on Valkey for this
workload — the capability filter runs in Rust after the FOR UPDATE
SKIP LOCKED scan (see `crates/ff-backend-postgres/src/attempt.rs`
module comment), avoiding the Lua fanout the Valkey path incurs for
cap-set matching. A cap routing workload that fits inside PG's lock
budget (≤ 50 concurrent workers) may be a genuine PG strong point.

### Scenario 6 — quorum-cancel fan-out (RFC-016 §4.2 SLO gate)

10 samples per size. `cancel_flow(CancelAll, NoWait)` timed end-to-end
against N pre-seeded sibling exec rows under one flow.

| n    | p50 (ms) | p95 (ms) | p99 (ms) | avg members cancelled | SLO verdict      |
| ---- | -------- | -------- | -------- | --------------------- | ---------------- |
| 10   | 3.87     | 51.74    | 67.83    | 10.0                  | characterization |
| **100** | **22.09** | **59.91** | **64.08** | **100.0**         | **PASS (≤500 ms)** |
| 1000 | 183.37   | 185.25   | 186.04   | 1000.0                | characterization |

**Ship SLO at n=100: PASS with ~7.8× margin** (p99 = 64.08 ms vs
500 ms target).

This is an approximation of the full AnyOf/Quorum + CancelRemaining
path — see Scenario 4 for why the flow-ingress FCALLs aren't yet a
peer on PG. `cancel_flow`'s SERIALIZABLE cascade is a conservative
upper bound on what the full path will cost once the ingress FCALLs
land, because the cascade does the heavy work (flipping every
non-terminal member + writing a completion outbox row per member in
one tx) regardless of whether the trigger was an explicit `cancel_flow`
or an AnyOf/Quorum-group satisfaction event.

Validity check: `avg_members_cancelled` is the number of exec rows
`cancel_flow` returned in its `Cancelled::member_execution_ids` vec.
All three points show it matching N exactly, which confirms the
cascade fired. (An earlier draft of this bench had a partition-key
mismatch that returned in sub-millisecond with zero members
cancelled; fixed by seeding into
`flow_partition(flow_id, PartitionConfig::default()).index`.)

## §3. Host tuning

Postgres's default `max_locks_per_transaction = 64` is insufficient
for 16 concurrent claim/complete transactions against a
256-partition ff_exec_core on this schema. The WAL logs show the
failure mode:

```
ERROR:  out of shared memory
HINT:  You might need to increase max_locks_per_transaction.
STATEMENT:  UPDATE ff_exec_core SET lifecycle_phase='terminal' ...
```

Bumping to 512 unblocks the bench matrix. This is a **real ops
concern**, not a bench artefact. Production Postgres deployments of
FlowFabric will need to set this value explicitly — the default won't
hold under sustained load. Candidates for v0.7 docs:

- `RELEASING.md` or the Postgres setup README: call out the tuning
  requirement with an `ALTER SYSTEM SET max_locks_per_transaction =
  512` one-liner.
- `ff-backend-postgres::connect` could emit a startup warning if
  `SHOW max_locks_per_transaction` returns < 256.

## §4. Ship-or-no-ship call for v0.7.0

**Recommendation: ship-blocked pending Phase 1 widening.** The
RFC-016 §4.2 SLO at n=100 passes with comfortable margin, so the
release-gate obligation inherited from v0.6 is met. However:

1. **Scenario 2 not measured.** The suspend/signal/resume roundtrip is
   a first-class Pattern-3 workload; shipping the PG backend without
   a live-bench number is premature.
2. **Scenarios 3/4 configs not aligned with Valkey.** The throughput
   gap on scenario 1 is real and known; scenarios 3 and 4 produce
   numbers the comparison can't honestly interpret without matching
   workload shapes or, in the case of 4, without the flow-ingress
   FCALL peers on PG.
3. **Host tuning is a documentation gap.** Shipping without calling
   out `max_locks_per_transaction` puts first-time PG operators on a
   sharp edge — they'll hit "out of shared memory" under any
   sustained 16+ worker load and have no pointer to the fix.

Concrete gates for v0.7.0:

- [ ] Scenario 2 ports cleanly or gets a scoped carve-out.
- [ ] Scenario 3 re-run at config parity with Valkey (same worker
      count, same duration) OR documented as characterization-only.
- [ ] Flow-ingress FCALL peers (`create_flow`,
      `stage_edge_dependency`, `apply_dependency_to_child`) land on
      PG, at which point scenario 4 + the quorum-cancel scenario
      switch to the real path.
- [ ] `max_locks_per_transaction = 512` guidance added to ops docs
      + startup warning in `PostgresBackend::connect`.

The PASS at n=100 lets v0.7.0 ship **if the carve-out above is
explicit in the release notes** — operators know what they're getting,
and the SLO-gated machinery works. My lean is ship-with-carveout, not
block.
